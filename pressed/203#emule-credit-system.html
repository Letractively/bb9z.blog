必要的约定及说明 [TOP]

术语 [TOP]

在本译文中，“credit modifier”，“ratio”，“score” 统一作信用评价量，简称：评分（主观视角，对于上传者，即我们）或得分（第三方视角，向我们请求上传的客户，上传受益者）。

得分为对方在我们这里的系数，越高则请求将越快得到满足，对客户来说则越好。

约定 [TOP]

因为原文对传输方向的描述比较混乱，对于未限定方向的传输，上传量统一指我们传输给其他的人，简写“UL”，下载量与之相对，简写“DL”。含糊不清的地方按个人理解转换，出错的话还请包涵。

传输量单位若未注明则为MB。

如未特别说明或未有限定，描述站在第一或第二视角，客户指第三方。

一些澄清 [TOP]

信用系统就是我们通常说的积分系统，主要用来回馈曾给予我们恩惠的客户，仅在独立的两个客户端间有效，而不是一对多或多对多的。

仅从加速下载的角度（而不是纯分享）来看，你不能通过选择合适的信用系统直接改善你的下载，而是先优化你的上传队列，通过更有效的上传来减少排队的等待时间，同时也需要对方采用比较合理的信用系统。DLP也可以理解为信用系统的补充规则，拒绝或减少给某些客户端（通常是不愿回传的）上传来优化上传队列，从而间接达到下载的提速。

信用系统 [TOP]

在你上传给客户前他们在队列中的等待时间由一个得分决定，而信用系统决定了这个分数。

官方信用系统 [TOP]

评分有两种计算方法：

  • 得分1 = 2×DL/UL
  • 得分2 = (DL + 2) 再开方

然后比较取较小的作为评分。

存在一些边界规则：

  • 当DL小于1MB时， 评分 = 1
  • 当UL为零时，评分 = 10
  • 评分的范围限制在1到10

Peace [TOP]

和官方的一样，只是代码更短小（但是更慢！）

Xman改进版 [TOP]

对已有信用系统的提升，它对给我们传输多的用户以奖励。这个系统考虑奖励因素。

另一方面，会在当前会话中对那些你传给他多，他传你少的客户以惩罚。

正面奖励的计算公式：
奖励系数=(DL - UL)/10-10/DL

评分最大为10（和官方的一样）。

示例 （大约一个区块（约9.28MB）的不同）:

如果你在会话中多给对方1个区块的上传并且完成除去回馈的下载/上传比较，那么他会得到-0.1的系数惩罚。
如果你在会话中多给对方2个区块的上传并且完成除去回馈的下载/上传比较，那么他会得到-0.2的系数惩罚。

Magic Angel [TOP]

基于官方系统，不同在于：

  • 当DL超过1.65MB时开始记录信用（官方为1MB）
  • 评分下限为0.1（官方为1），上限为50（官方为10）

Magic Angel+ [TOP]

Magic Angel + 信用系统基于Magic Angel 信用系统，但DL多于UL时，评分会有些增加。

当客户的上传与下载之差：

< 7MB 评分+ 0.3
< 15MB 评分+ 1.0
< 30MB 评分+ 2.0
>= 30MB 评分+ 3.0

Ratio [TOP]

基于官方系统，评分最低为0.1而不是1。

对上传者有×2，×4，×16倍的奖励。

Ratio——Stulle的实现 [TOP]

代码来源: Stullemule 3.6 src - ClientCredits.cpp - 代码位置: case CS_RATIO: // RT.10a mod Credit

与它的名字相反，不是基于上传/下载比，而是上传、下载量和他们间的差值。

没有简单的唯一表达式。

默认评分=1。

对零（小于1MB）上传者逐步惩罚；
对未得到回报的上传者（总量少于1MB）逐步提高评分。

上传在 1~9MB的客户获得最低为DL/9的信用；
上传大于9MB的客户获得最低为0.7 + sqrt ( DL +1 ) / 10的信用，
例如：上传99MB获得1.7的信用。

通过平衡的信用系数按比例增加到 sqrt ( DL + 1)；

信用进一步的增减由上传减下载的差决定；

评价系数的值无限制。

图例和公式：

DL =总计下载MB ( to me )
UL =总计上传MB ( to him )

未认证用户的得分为1；
新客户（UL和DL不大于1MB）的得分为1；

DL<=1且UL>1（几乎零上传者）则评分为1/sqrt(UL)；
DL>1且UL<=1（应当被回馈）则评分为DL；

若DL和UL均大于1MB，定义： 基础系数= sqrt(DL+1)， 修正系数= sqrt(abs(DL-UL))。

  • 当DL>UL，评分= 基础系数 + 修正系数
  • 当UL-DL<= 1，评分= 基础系数
  • 当UL-DL>1且DL>=9，评分= max ( 0.7 + (基础系数/ 10), 基础系数/ 修正系数)
  • 当UL-DL>1且DL<9且基础系数<修正系数，评分=DL / 9
  • 当UL-DL>1且DL<9且基础系数>=修正系数，评分= 基础系数/ 修正系数

Neo [TOP]

默认分数为1.0。上传下载量以字节（Bytes）计，在任何情况下如果客户没有我们需要的文件则计1.0分。如果客户没有良好的用户标识（译注：subuser Identification，即平常说的认证用户 ）则只计算本次会话的传输。

如果UL大于DL，公式：
评分=( (9500KB×2(未认证用户) or 4(认证用户))/(UL-DL) )^2
值的范围限制在0.1至1.0

如果UL小于DL，公式：
评分=( (DL-UL)/(9500KB*1(未认证用户) or 2(认证用户)) )^2.
值的范围限制在1.0至2.0

Pawcio [TOP]

值的范围从1.0到100.0；

乘数为3（而不是2）——评分=3×DL/UL；

对于新人（上传、下载少于1MB的），乘数为3（而不是1） ；

如果接受某人超过1MB但未回馈给他任何内容，得分为100.0。

如果某客户传给你很多：

  • 如果你获得了100MB，那么用户得分为50.0直到你回馈108MB；
  • 如果你获得了50MB，那么用户得分为25.0直到你回馈55MB；
  • 如果你获得了25MB，那么用户得分为12.0直到你回馈28MB；
  • 如果你获得了15MB，那么用户得分为5.0直到你回馈12MB。

Fine [TOP]

对下载部分文件而未回传的客户以惩罚。

词条：Fine CS（http://wiki.emule-web.de/index.php/Fine_CS）

SWAT [TOP]

评分有两种计算方法：

  • 得分1 = 2.2×DL/UL
  • 得分2 = sqrt(DL+2)

然后比较取较小的作为评分。

存在一些边界规则：

  • 当DL<1MB（精确的！）时，评分 =1
  • 当UL=0时，评分=10
  • 范围限制在1至100。

Sivka [TOP]

开启加密但未认证用户（IS_IDNEEDED）的评分为0.75 ；
认证失败用户（failed ident）的评分为0.5；
不良用户（Bad client）的评分为0。

定义传输差值=DL-UL

如果传输差值大于1GB，评分为32 ；
如果传输差值介于0至1GB，评分=sqrt(差值)
其他情况，评分为1。

Eastshare [TOP]

  • 基本评分：认证用户＝100，不支持用户＝80，认证失败＝0
  • 评分最低为10，最高为5000
  • 每上传1MB，评分＋6； 每下载1MB，评分－2
  • 若DL > 1MB，评分＋100
  • 若评分 < 50，DL> 1M，评分为50

（译注：本节描述与[1]，[2]互不相同）

Lovelace [TOP]

评分=100*( (1-1/(1+exp((3*DL^2-UL^2)/1000)))^6.6667 )
起评分为1，最小为0.1，最大为100，
比率：1:1.5，只有一个公式

信用窃取者（Credit Thefts）不会得到任何信用，只有使用安全认证的客户才能得到100的系数，其他为10。和原始信用系统相比，
信用评价更偏重（传输）差而不是（上传/下载）比率。使用原始系统时，你能在生成新userhash后的短时间内获得较好的积分；
而使用新积分系统时，如果你已经上传了很多（并且未通过更换userhash作弊），可以更快获得较好的信用值。

旧系统（对上传者而言）：

  • UL=5MB，DL=5MB，则下载得分为2，
    再增加5MB的上传，可获得4的下载得分。
  • UL=10MB，DL=10MB，则下载得分为2，
    再增加5MB的上传，可获得3的下载得分。

对于同等数量的上传你获得更少的得分（-25%）。

新系统（对上传者而言）：

  • UL=5MB，DL=5MB，则下载得分为1.16，
    再增加5MB的上传，可获得2.31的下载得分。
  • UL=10MB，DL=10MB，则下载得分为1.85，
    再增加5MB的上传，可获得5.09的下载得分。

对于同等数量的上传你获得更多的得分（+120%），因为之前你已经上传了特定的数量。

ClientAnalyzer [TOP]

（译注：用户分析，简称“CA”）

起评分为1，最小值为0.01，最大值为10

客户得分的评价基于他们的行为，比如：

一个客户知道另一个客户多长时间（对保持userhash的用户给予每周的奖励），上传/下载比，上传/下载的种类（已完成/部分/稀有），反吸血设置（Anti-Leecher options），昵称窃取（Nickthieves）检测，客户端标识窃取（Modthieves）检测，文件伪造者（Filefakers）检测，滥发消息者（Spammers）检测，XS-Exploiter检测，被考虑的Hammerer（hammerer are taken into account），平均重新请求时间（avg reask time）等。

TK4 Credit System [TOP]

（译注：本节中出现的原始单位为Mb、Gb，疑为MB、GB）

TK4 Mod 1.5f的信用系统，基于eMule 47a. C++源代码，完整的文本可见：TK4 Mod - Credit systems（http://emuletk4mod.sourceforge.net/TK4_Mod_Creditsystem.htm）

如果你分享已经完成的文件，因为你已经拥有了这部分文件，请求这些文件的客户不会回传给你数据，所以在TK4系统中他们的信用评分不变。
如果你正下载文件并且某人从你拥有的那部分获得数据，他们将服从该积分系统，并且由他们给予的数量决定他们的评分有多少降低。
在任何时候，向你传输数据的任何人的评分将上升。

积分系统大体像这样工作：

起评分=10，
补贴=DL/4+1

如果UL>(DL+补贴+1)（他们拿的比给的加上一定量还多），并且那个文件（被请求的文件）是我们正在下载的：

  • 如果(UL-补贴)介于1到81MB
    评分乘数的范围为9到1
    公式：10 - sqrt((UL - 补贴)-DL)
  • 如果(UL-补贴)大于81MB
    评分乘数的范围为1到0.14
    公式：9 / sqrt((UL - 补贴) -DL)
  • 如果DL>UL（他们给的比拿的多）：
    评分乘数的范围为10到342
    公式：10+log(2.72+(DL-UL)*4)+(UL/12)

乘数没有绝对的范围，但是我报了一个大于0.14小于342的范围（假设最大的差值为4GB）用10作为起始值。
认证失败的客户起始为10，但不会上升。

Reverse [TOP]

又称“credit shaping”。

不良特性（见：http://wiki.emule-web.de/index.php/Bad_feature）

  • 给予在你下载队列中客户的评分以10倍的评分增量（你很快会等待一些（请求）从他们那）
  • 对无需要部分和满队列客户以双倍的判罚（当前无需求但也许将来有，当所有其他来源有至少5的得分时上传给他们）
  • 给予少量评分的提高如果预计的等待时间较短（这将会在下个版本中生效）
  • 给等待时间以少量评分的提高（最少的），仅仅出于公平。

事实上，我们的客户端会按下面顺序满足其他客户：

  • 我们试图下载的并且我们没有向其上传过任何数据的客户（他们将获得最大得分的1.1MB），在他们队列中等待时间短者优先；
  • 已经回馈过数据且我们正试图下载的客户，比率直到5倍；
  • 我们正试图下载的但是处于无需要部分或队列已满状态的得到1.1MB；
  • 所有其他我们正试图下载的且已经上传过的，比率直到10倍；
  • 所有有着最大评分的客户将以先到先得的原则回馈。
